一、创建型模式

1、单例模式

Singleton

一个类只有一个实例，且该类能自行创建这个实例的一种模式

特点

  • 私有的构造方法，外部不能直接创建该类的实例

应用

  • 某类只要求生成一个对象，对象需要被共享的场合
  • 当某类需要频繁实例化，而创建的对象又频繁被销毁的时候

​ 如多线程的线程池、tomcat连接池等

1.1 懒汉式

  • 多次调用都会调用Synchronized锁，性能较低。

1.2 饿汉式

实例在初始化的时候已经创建，JVM在加载类的时候是单线程的，由于实例被static修饰，故没有线程安全的问题

  • final–防止子类继承覆盖，破坏单例
  • 构造方法私有，防止创建新的实例，但不能防止反射创建新的实例
  • static修饰在加载类时创建，JVM在加载类的时候是单线程，可以保证是线程安全的。
  • 方法封装提供一些懒惰的初始化，提供泛型的支持

1.3 双检锁式

特点是在synchronized关键字内外都加了一层 if 条件判断

既保证了线程安全，又比直接上锁提高了执行效率，还节省了内存空间

  • 双重if，防止并发条件下，单例对象不会被重复创建。

1.4 静态内部类

实例化设置为静态内部类的一个静态属性。

只适用于静态域的情况，双检锁方式可在实例域需要延迟初始化时使用。

  • 第一次使用时，才会触发类加载操作的，故属于懒汉式

  • 类加载时JVM是单线程的，无线程安全问题。

1.5 枚举单例

可为饿汉式的改进

自动支持序列化机制，绝对防止多次实例化。

  • 枚举实例是静态的，创建时线程安全
  • 不能被反射破坏单例
  • 枚举类默认实现序列化接口，不会被反序列破坏单例
  • 属于饿汉式单例

3、工厂模式

3.1 简单工厂模式

  • 将全部创建逻辑集中到了一个工厂类中；
  • 它所能创建的类只能是事先考虑到的，如果需要添加新的类，则就需要改变工厂类了，而增加、删除某个子类对象的创建都需要打开简单工厂类来进行修改违背了开闭原则。
  • 又称为静态工厂模式

3.2 工厂方法模式

  • 用户只需要知道具体工厂的名称就可得到所要的产品，无须知道产品的具体创建过程；
  • 在系统增加新的产品时只需要添加具体产品类和对应的具体工厂类，无须对原工厂进行任何修改，满足开闭原则
  1. 抽象工厂（Abstract Factory）：提供了创建产品的接口，调用者通过它访问具体工厂的工厂方法 newProduct() 来创建产品。
  2. 具体工厂（ConcreteFactory）：主要是实现抽象工厂中的抽象方法，完成具体产品的创建。
  3. 抽象产品（Product）：定义了产品的规范，描述了产品的主要特性和功能。
  4. 具体产品（ConcreteProduct）：实现了抽象产品角色所定义的接口，由具体工厂来创建，它同具体工厂之间一一对应。

应用

  • 客户只知道创建产品的工厂名，而不知道具体的产品名。如 TCL 电视工厂、海信电视工厂等。
  • 创建对象的任务由多个具体子工厂中的某一个完成，而抽象工厂只提供创建产品的接口。
  • 客户不关心创建产品的细节，只关心产品的品牌。

3.3 抽象工厂模式

抽象工厂模式将考虑多等级产品的生产，将同一个具体工厂所生产的位于不同等级的一组产品称为一个产品族

应用

  • 系统中有多个产品族，每个具体工厂创建同一族但属于不同等级结构的产品。
  • 系统一次只可能消费其中某一族产品，即同族的产品一起使用。

优点

  • 可以在类的内部对产品族中相关联的多等级产品共同管理，而不必专门引入多个新的类来进行管理。
  • 当增加一个新的产品族时不需要修改原代码，满足开闭原则。
  • 但当产品族中需要增加一个新种类的产品时，则所有的工厂类都需要进行修改，不满足开闭原则。

Proxy

一、概述

  • Java Data Base Connectivity（java数据库连接）是一种用于执行SQL语句的JavaAPI

  • 它由一组类和接口组成

  • JDBC API使用java的反射机制来实现java程序和JDBC驱动的松耦合

二、JDBC API

（1）接口

  • DriverManager驱动管理类，它可以创建连接，本身就是一个创建Connection的工厂

  • java.sql.Driver 主要功能是描述驱动程序，提供驱动信息，最重要实现数据库连接的抽象

  • java.sql.Connection 是数据库jdbc连接的抽象层，主要生成SQL并发送给数据库

  • ResultSet接口（结果集），用来接收select语句返回的查询结果（true/false(query)、num(Update)）

（2）Statement接口

​ 编译语句 java.sql.Statement

  • 由createStatement创建

​ 预编译 java.sql.PreparedStatement

  • 继承自Statement接口，有preparedStatement创建，用于发送含有一个或多个参数的SQL语句
  • 效率比Statement高，并且可以防止SQL注入

​ 数据库存储过程 java.sql.CallableStatement

  • 继承自PrepareStament接口，由prepareCall创建

（3）Statement方法

  • execute: 返回是否有结果集

  • excuteQuery：运行select语句，返回Result结果集合

  • excuteUpdate：运行insert/update/delete操作，返回更新的行数

  • addBatch(String sql): 将多条sql语句放到一个批处理中

  • executeBatch() : 向数据库发送一批sql语句

  • set(int colunmIndex,Type val) ：为PreparedStatement回填参数值

  • getString(int index) getString(String columnIndex)

1、maven概述

1.1 俩个核心

  • 依赖管理：对jar包统一管理

  • 项目构建：项目在编译完成后，对项目进行编译、测试、打包、部署等一系列操作通过命令完成

1.2 仓库类型
  • 本地仓库：在个人计算机上
  • 私服：存在于局域网内一台服务器，存jar包
  • 中央仓库：在互联网上，存放基本上所有开源的jar包(repo1.maven.org)

一个jar包一旦被下载过，就会被Maven自动缓存在本地目录

2、项目结构

  • pom.xml-maven项目核心配置文件project object model

    • groupId-类似java的包名，往往是公司名称
    • artifactId-构建名
    • version- 构建版本号

    作为maven项目的唯一标识。

    引用其他第三方库时，也是通过这3个变量确定。

  • main

    • java-java文件
    • resources-资源配置文件

  • test-单元测试

  • target-编译打包生成的文件

3、依赖关系

compile、test、runtime、provided

scope 说明 示例

runtime 编译时不需要，但运行时需要用到 mysql/jdbc驱动

4、生命周期

lifecycle-phase-goal

Maven的生命周期lifecycle由一系列阶段phase构成

  • mvn这个命令时，后面的参数是phase，Maven自动根据生命周期运行到指定的phase。

**生命周期default:**validate - initialize - generate-sources - ……..

生命周期clean：preclean - clean - postclean

  • 运行mvn package，Maven就会执行default生命周期，它会从开始一直运行到package这个phase为止：

  • 运行mvn compile，Maven也会执行default生命周期，但这次它只会运行到compile

常见的命令

  • mvn clean：清理所有生成的class和jar；
  • mvn clean compile：先清理，再执行到compile；
  • mvn clean test：先清理，再执行到test，因为执行test前必须执行compile，所以这里不必指定compile；
  • mvn clean package：先清理，再执行到package。

Goal

执行一个phase又会触发一个或多个goal

goal是最小任务单元

goal的命名总是abc:xyz这种形式 plugins:goal

执行的Phase 对应执行的Goal
compile compiler:compile
test compiler:testCompile surefire:test

5、插件

执行每个phase，都是通过某个插件plugin来执行

Maven本身其实并不知道如何执行compile，它只是负责找到对应的compiler插件，然后执行默认的compiler:compile这个goal来完成编译。
